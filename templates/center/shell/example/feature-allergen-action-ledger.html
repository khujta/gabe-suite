Docs The feature card

the entity in one handle, one analogy and one constraint box

Authored in cards/allergen.md — the only hand-written source on this page. Everything counted elsewhere is read from the codebase at build time.

“"Never show a recipe that can hurt this user — enforced in the query, not the UI."”

ISthe SC-03 safety guarantee — a user with a declared allergen never sees a violating recipe on any list, explore, pantry-match, or meal-plan screen.
IS NOTa soft "fuera de tu dieta" preference badge (that is the DIET/preferences half, D17) — allergens are a hard exclusion, diets are a soft label.
DECIDESwhether a recipe row is EXCLUDED from a result set (allergens) vs merely FLAGGED (diet preferences).

a bouncer with two lists — a name list (the recipe's self-reported allergen tags) and a face-recognition camera (its ingredients matched against the allergen catalog). You are turned away if EITHER flags you.

AnalogyIn this codebase
name listservices/recipes.py:194
face cameraservices/recipes.py:224

Easy to confuse:

  • the avoid/exclude browse facets (No usar técnica/textura/temperatura) — those are per-search PREFERENCES, not the allergen wall.
  • services/safety.py — that is the SOFT food-safety avisos layer (botulism, salmonella), a different domain from the allergen exclusion.

Where the analogy stops working: the bouncer analogy breaks where the two lists disagree on identity — the SQL join matches the RAW ingredient code, while the point-read assert resolves aliases first, so an aliased ingredient is a "face" only the assert recognizes.

Full card — what & why, for whom, flows, is / is not

What & Why

Gustify's core safety promise (SCOPE SC-03): a user with any tag in DietaryProfile.allergens[] never sees a recipe containing that ingredient — or violating that diet — on any screen, enforced at the database query level so no UI-layer gap can leak a violating recipe. Allergens are a hard exclusion; diet preferences[] are soft-filtered (surface with a "fuera de tu dieta" badge — D17's binary two-list model, superseding D16's three-level proposal).

The wall is positive-evidence: a recipe is dropped only when a signal AFFIRMATIVELY flags an allergen — self-reported restriction_codes (exact JSON-array token, #17) OR an ingredient that maps to the allergen in the catalog (#38/G18 ingredient-derived closure, D67/D69). Excluded on either signal. Because it excludes on evidence rather than defaulting closed, its guarantee is contingent on catalog completeness — the deploy-time boot-reconcile/seed is what keeps restriction_codes fresh and ingredient_restrictions populated (the machinery that closed the earlier fish-allergen leak).

For Whom

  • The family cook and every household co-cook — the person who declared an allergy trusts that no suggestion, list, or plan can surface a dish that violates it. This is the one promise the app cannot break quietly.
  • The apprentice / secondary cook — shares the household allergen constraints; every screen they see is filtered the same way.
  • Whoever prices safety risk — the register on the Risk tab names exactly where the guarantee is contingent (catalog completeness) and where two enforcement mechanisms disagree (alias resolution), so the residual risk is legible, not assumed away.

Flows

  • declare → the user sets allergens in Ajustes / Alergias (cap of 5, MAX_ALLERGENS) or the setup wizard; the heavy danger-toned confirm gate guards REMOVAL (H11), the redo wizard preloads declared allergens so re-onboarding can never silently wipe them to [] (C2), and the R8 allowlist filters any non-canonical code before the write.
  • resolve → load_user_allergens (services/recipes.py:1409) reads the profile per request at query-build; the dialect is resolved from the live bind, so the same builder works on SQLite and Postgres.
  • exclude ★ → every list surface (browse, explore, pantry cookability, meal-plan generation, favorites, saved-modes, planned, cooked, pantry-match) funnels through _apply_recipe_filters (services/recipes.py:654-669), which ANDs both allergen condition sets under if user_allergens: — re-applied on every request, so a once-safe favorite that becomes unsafe is excluded on the next fetch.
  • backstop → the by-id detail, plan, and cook-start point reads call assert_recipe_allergen_safe (services/recipes.py:1226) → 403 before building the body; this assert resolves aliases via the resolution snapshot, making it stricter than the browse SQL join.
  • heal → deploy-time reconcile_recipe_restrictions re-derives every slugged recipe's restriction_codes and upserts catalog-delta ingredient_restrictions pairs, keeping the positive-evidence wall's inputs complete.

Is

  • An always-on SQL WHERE exclusion on every browse/list/suggest surface — one shared builder, not a per-endpoint reimplementation.
  • Two-signal: self-report tags (exact token, NULL-guarded) UNION ingredient-derived (NOT EXISTS join). Excluded on either.
  • Layered: browse = SQL; point reads (detail/plan/cook) = a stricter alias-resolving 403 assert.
  • Catalog-contingent: a positive-evidence filter kept honest by boot-reconcile/seed completeness.
  • Fed by declaration paths that are themselves guarded (cap=5, R8 allowlist, redo-preserve C2, heavy-confirm-on-removal H11).

Is Not

  • Not a single SQL path. SC-03 is delivered by TWO mechanisms — the browse SQL join and the point-read app-layer 403 assert — that derive their answer along different code paths (verified single_path=false).
  • Not fail-closed. It excludes on positive evidence; a recipe with NULL/omitting restriction_codes AND no catalogued ingredient match is served. The guarantee rests on catalog completeness, not a closed default.
  • Not the diet/preference filter. Diet preferences[] are soft ("fuera de tu dieta" badge); only allergens[] are hard-excluded (D17).
  • Not the avoid/exclude browse facets (técnica/textura/temperatura/equipo) — those are per-search user preferences, not the safety wall.
  • Not the food-safety avisos (services/safety.py) — that is a distinct advisory-warning domain (botulism/salmonella).
  • Not proven by property/fuzz. The SC-03/REQ-07 "randomly-generated profiles × catalog, zero leaks" acceptance signal is documented but unimplemented; coverage is hand-picked fixtures.

About this section

The wall lives in services/recipes.py: the two condition builders (allergen_filter_conditions L194, ingredient_allergen_exclusions L224) composed in _apply_recipe_filters L654-669, the per-request resolver load_user_allergens L1409, and the point-read assert assert_recipe_allergen_safe L1226. services/safety.py is the adjacent soft-avisos layer. services/restriction_seed.py + services/reconciliation.py keep the catalog (the wall's evidence base) complete. Declaration surfaces are the settings/setup routes + the web settings safety components. The data model is the dietary profile + the ingredient→restriction catalog.
