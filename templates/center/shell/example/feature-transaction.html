“One plate on the press; bound sheets freeze”

ISThe ledger spine — every way a purchase enters (scan, draft, direct create), lives (list, detail, edit, flags, images), and leaves (a 90-day delete window) the ledger.
IS NOTThe scan pipeline or the reconciliation verdict. Both hand off to this entity; neither lives here.
DECIDESWho loses the right to edit or delete, and exactly when.

A printing press. One plate sits in the bed at a time — you keep re-inking it while the design changes. One typesetting yields many impressions on different paper sizes. Once a sheet is bound into the ledger book that impression can't be re-inked, though you may still pencil a note in its margin — and after ninety days the binding itself is sealed.

Easy to confuse:

  • draft slot vs transaction — the slot is ONE mutable row per user; the transaction is the permanent record. Commit crosses that line and clears the slot.
  • shared-lock vs matched-lock vs window-lock — three different owners of "no": a group depends on it (D74), the statement is source of truth (P117), or the settled period is (90-day window).
  • manage vs embed — one renderer, two chromes; not two components.

Where the analogy stops working: When reasoning about multi-device editing, a plate is physically one object; the draft slot is reachable from several devices at once, and the UNIQUE constraint — not physics — is what arbitrates.

What & Why

Transaction is the T1 ledger entity — the permanent record of a purchase. This section covers the WHOLE entity, not just its recent redesign: three ways in, one life, guarded ways out.

Ways in. A scan hands off extraction into the draft payload (POST /scans?target=draft — nothing persists as a transaction until commit); manual entry lives in the same single draft slot (one per user, DB-enforced, autosaved, survives reload and device switch); and POST /transactions creates directly — the path the scan's persist mode and the mobile app use. All three converge on the same create machinery: validation, the USD FX shadow (amount_usd_minor + fx_rate_to_usd captured at write), card-alias checks, and — when the source was a scan — the AI-cost telemetry columns (llm_tokens_in/out, llm_cost_usd, latency and queue timings).

Its life. One TransactionList renderer serves the Compras counter and the Historial embed; the detail groups items into L3 "familia" tone-cards with inline per-field PATCH; receipt images stream from the DB (compressed bytes, scope-checked, pre-migration rows honestly 404); per-item flags survive every lock; recurrence is a first-class dimension (none | fixed_term | recurring | unknown, with term_current/term_total for fixed terms — "cuota 3 de 12").

Ways out — and refusals. Sharing (D74) or a statement match (P117) freezes content while flags stay live, and GET /{id}/lock-links names exactly WHAT locks a row. Deletion works only inside the 90-day window (transaction_delete_window_days, default 90) — past it, a 409 explains that settled periods' statistics can't shift underneath the user. Batch update (merchant + store category only) and batch delete follow whole-batch rejection: if ANY selected row is statement-matched, the whole batch 409s rather than silently partial-applying.

One clipboard on the counter, one drawer for the receipt you're still writing, a wax seal on pages someone else depends on — and the whole book bound shut after ninety days.

For Whom

The person capturing a purchase — by camera, by hand, or from the phone — who expects the half-finished one to still be there after a reload, and a plain reason whenever the app says no: why this field is frozen, why this receipt can't be deleted, why the whole batch refused. The finance-minded user who expects USD-shadow totals and "cuota 3 de 12" to be first-class. And the developer: one list renderer, one create machinery, and invariants enforced by the database and named 409s rather than by hope.

Flows

  • scan → POST /scans?target=draft → worker writes extraction into the draft payload → review → commit
  • manual → the single draft slot (GET/PUT/DELETE /transactions/draft) → debounced autosave → POST /commit → the slot clears
  • direct → POST /transactions (scan persist mode, mobile) → same validation + FX shadow + alias checks
  • browse → Compras counter (variant="manage") or Historial embed (variant="embed") → one renderer
  • open → detail → familia item groups · inline per-field PATCH · authed image fetch (GET /{id}/images/{image_id})
  • flag → PUT /{id}/items/{item_id}/flags — works even on locked rows
  • lock → shared (D74) or statement-matched (P117) → content frozen · GET /{id}/lock-links names the owner
  • delete → DELETE /{id} inside the 90-day window · 409 past it · batch delete rejects whole-batch on any matched row
  • batch → select in the list → POST /batch-update (merchant, store category) / POST /batch-delete

Is

  • one draft slot per user — transaction_drafts, DB-enforced by UniqueConstraint("user_id", name="uq_transaction_drafts_user") (D110); GET/PUT/DELETE "" + POST /commit under prefix="/transactions/draft"; PUT is an idempotent last-writer-wins upsert
  • three entry paths, one machinery — draft commit and POST /transactions share create logic: validation, FX/USD shadow (amount_usd_minor, fx_rate_to_usd, fx_captured_at), card-alias resolution, recurrence fields
  • recurrence as data — recurrence_kind ∈ none|fixed_term|recurring|unknown + interval, term_current/term_total, label, source and confidence (scan-detected or user-set, recurrence_user_edited_at)
  • receipt images in the DB — compressed bytes streamed by GET /{id}/images/{image_id}, scope-checked via the owning transaction; pre-migration rows carry no bytes → honest 404; thumbnails flagged per image
  • one list renderer — TransactionList (+ co-located TransactionRow), variant manage | embed; batch selection prunes to still-present ids (C1349)
  • the 90-day delete window — settings.transaction_delete_window_days (default 90, configurable); DELETE /{id} past the cutoff 409s with the named rule; batch delete mirrors D74: whole-batch 409 over silent partials
  • batch surface — POST /batch-update changes exactly merchant and store_category_id; POST /batch-delete; both scope-checked
  • lock provenance — GET /{id}/lock-links: the NEWEST MATCHED verdict's statement line + the group copies the caller can see; TxnLockPopup degrades to its base form if the fetch is empty
  • AI-cost telemetry on the row — llm_tokens_in/out, llm_cost_usd, scan_duration_ms, llm_latency_ms, queue_wait_ms (the T3 funnel reads these; they live here)
  • store-category provenance — id + source + confidence + mapping id + user_edited_at, so a user correction is never silently overwritten

Is Not

  • not the scan pipeline itself — extraction, quota, review levels and the worker are the scan-receipt section; this entity owns the draft handoff and the telemetry columns the scan writes
  • not the statement side — reconciliation verdicts belong to statement; here a MATCHED verdict is an input to the lock
  • not analytics — the drill from a category into these rows is analytics-drilldown
  • not the card-alias rules — payment set/clear reuses card_alias_id; the alias invariants are their own section
  • not the mobile UI story — mobile consumes the same API (TransactionsScreen, TransactionDetailScreen); no Maestro flow is attached here yet
