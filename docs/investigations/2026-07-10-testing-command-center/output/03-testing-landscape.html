2 · What we run, and why — the kitchen map

Gustify is a cooking app, so: the backend suite tastes every sauce in the prep kitchen; the journeys are full dress-rehearsal dinner services; docker-smoke turns the key on the actual food truck. Badge = test kind (these letter badges become PixelLab icons once you pick the visual direction).

Static gates

the health inspector at the door — nothing runs, everything is checked

The allergy thread: the OpenAPI freeze gate blocks any commit that renames the restriction_codes field every allergen filter depends on — the contract breaks loudly at commit time, not quietly at runtime.

ruff · pyright(strict) · eslint · tsc · gitleaks · legacy-import blocker · OpenAPI freeze (additive-only contract) · dialect-portability walk.

  • Guards: style/type drift, leaked secrets, architecture boundaries, breaking API contract changes, the SQLite-vs-Postgres default bug class (PENDING #28 burned us once)
  • When: pre-commit + every CI push · lands in: terminal/CI logs

pytest — backend (1,095)

tasting every ingredient and sauce in the prep kitchen

ASGI-level: real routing + dependency injection, fake network. Runs 3 ways: SQLite on every push (fast loop), full suite on Postgres 16 at the main gate, and docker-smoke boots the real production image against Postgres exactly as Railway does (~1 min).

The allergy thread: ~130 tests assert a milk-allergic user’s recipe list NEVER contains a milk-bearing recipe — exact-token JSONB matching (no substring false-negatives), NULL-guards so unrestricted recipes never vanish, proven on both SQLite and real Postgres.

  • Highlights: allergen/exclusion filters (the safety core) · migrations tested up and down per revision · SSE+WebSocket same-event-vocabulary contract · XFF-spoof defense (20 tests) · Gemini error taxonomy (SDK mocked; a fixture prevents any live call)
  • Lands in: CI logs; LEDGER prose totals

vitest — web (~408)

checking the recipe cards and mise en place of the dining room

Mostly pure model/logic (filters, ranges, cooking-session state, pantry) + the i18n 100% key-parity gate (es-CL ↔ en-US identical keys, matching placeholders, no empty strings — one missing translation turns CI red).

The allergy thread: the dietary-sync model tests assert the milk restriction chosen at setup propagates into every filter panel’s defaults — the UI can’t silently forget it.

  • When: every web push · lands in: CI logs

Storybook-as-tests (107 stories)

the plating rehearsal that exists but nobody schedules

The allergy thread: the filter-panel stories render the active-restriction states in a real headless browser — the visual warning chip, not just the logic behind it.

A second vitest project runs every story as a render/interaction test in headless Chromium. CI runs only the unit project — this one is manual. Wires into the LOCAL loop under D10 — the CI pipeline must not grow (round-2 constraint).

Playwright journeys (19 specs / ~86 tests, 8 groups)

full dress-rehearsal dinner service, one table at a time

Real browser → local web client → deployed staging-e2e API: real Firebase login, real Postgres writes, deterministic mock Gemini. Single worker (all specs share one test user). Every step screenshots into the Evidence system; capture mode records cursor video.

The allergy thread: the SAFE journey signs in as the fish-allergic test user (R8), triggers the live safety warning (“Avisos de seguridad”) and asserts the fish-bearing recipe is excluded from the real UI — with a numbered screenshot at every step. This is literally PLAN.json phase 1’s proof.

  • Groups: cooking·storage·browse·i18n·profile·settings·core·api-loop
  • When: cook-state auto-gates staging pushes; the rest run on-demand ([e2e:group] commit token or manual dispatch)
  • Lands in: numbered screenshots + manifest + run-status.json (the one durable test-result file in the repo)

5 · What's missing, ranked by the nature of the problem

Gustify is an allergen-safety-relevant, data-stateful, AI-generative, bilingual app, solo-operated at MVP maturity. That ranking, not a generic checklist, decides what's actually missing:

1 · Gemini output safety harness RETIRED by operator ruling (round 2)

Operator decision 2026-07-10: user-level Gemini recipe generation is being deprecated — remaining pieces in the app will be removed. All recipes will be generated in-house (a cloud session / project skill in gustify), not by a runtime provider. The runtime golden-harness proposal (old D9) dies with it.

What survives, transformed: the safety invariant moves from runtime to the pipeline — every in-house-generated recipe entering the catalog must pass the same R8 exclusion/allergen filters the catalog passes. Validation form is TBD by the operator's own words ("maybe another kind of validation there — not sure"). Parked as an open item on the in-house generation work, not a center decision. One residual for the center: track the deprecation itself (the Gemini test files and mock lane shrink as pieces are removed — the center should show that curve, not hide it).

2 · 107 Storybook tests that never run ✓ D10 agreed — LOCAL only

They exist, they work, nothing invokes them. Operator constraint (round 2): the GitHub Actions pipeline must NOT grow — private repo, CI-minutes quota exhausted before. So they wire into the local loop instead: a local runner step (same pattern as run-journeys.sh) + the center displays their last local run. Zero CI minutes.

3 · Coverage — which sauces has nobody tasted? ✓ D1 granted

Zero coverage tooling today. Lands as normal dev-deps in the spike; the center gets its coverage dimension.

4 · Dependency audit

No pip-audit / npm audit / Dependabot anywhere. One cheap CI step each (→ D10).

5 · Accessibility baseline

Nothing automated. A cooking app is used mid-task — wet hands, glare, timers running. An axe pass riding the existing journey specs on the 4–5 core screens is low-cost and honest about this app's real usage context.

6 · The invisible manual suites

Mockups, layout ×3, token-class-coverage are manual by design or by accident — today you can't even see when they last ran. Minimum fix: the center shows last-run dates, so "manual" stops meaning "invisible".

Consciously OUT at MVP (named, not missing)

Performance/load — PENDING #23/#25 already hold the "when latency signals appear" triggers. Mutation testing, chaos/fault injection — enterprise-tier practices; wrong maturity. Pixel-diff visual regression — the Evidence Doctrine's side-by-side proof + layout policy checks carry MVP; revisit if UI churn starts breaking things silently.

D3 · History custody — the ship's logbook decision

Static pages have no memory. For the center to show trends ("this spec went red 3 runs ago", sparklines, the fading greens above), every regeneration appends one line — date, source, totals — to a history file the next build reads. The decision is only: where does that logbook live?

Committed to git ✓ recommended

The logbook in the ship's safe. Survives machine changes; CI and local write to one shared memory; the deployed site keeps its history; a fresh clone shows trends immediately.

Cost: ~200 bytes per checkpoint. Capped at the last ~50 runs ≈ 10 KB total — the oldest line is dropped as a new one lands. Needs one named exemption in the docs-budget gate.

Gitignored (local only)

Sticky notes on one sailor's bunk. Zero repo noise — but CI's runs and yours never merge, a fresh clone has amnesia, the deployed site shows no trends.

And we've been burned: the Evidence Doctrine records that a gitignored run-status manifest was once deleted by a background cleanup and every doc page reverted to "not run". The manifest IS the memory — that lesson is literally written into the suite's conventions (§3).

Impact if chosen wrong: gitignored → you rediscover the deletion incident and trends silently lie (local-only view). Committed uncapped → the file bloats every commit. Committed + capped has no discovered failure mode. Awaiting your call — recommendation: committed, capped at 50.

D6 & D7 · The two Wave-2 flags — nothing is built today, you're only agreeing to timing

D6 — "proof" becomes an address instead of a sentence

Today

proof: "web-journey: triggered warning shows 'Avisos de seguridad'"

A sentence. A human can read it; the board can only print it. Nothing machine-readable names the spec file or the screenshot folder, so the board card is a dead end — you read, then go hunting.

After D6 (Wave-2)

proof: {spec: "tests/web-e2e/web-journey-….spec.ts", artifacts: "proof/safety/", verified: 2026-07-09}

An address. Every board card becomes a door: click → the spec, the screenshots, the run that proved it. This is what extends "everything deep-linked" to the KDBP layer.

Why it must wait for Wave-2: five suite skills (gabe-plan/execute/review/commit/push) all write PLAN.json — changing the proof schema means changing all five writers + validators together, in the suite repo, with its own careful pass. Exactly the kind of change this research phase is forbidden from making. Impact of never doing it: the board stays "a prettier PLAN.json" — readable, not navigable. The only decision now: keep it on the Wave-2 list. (Recommended: yes.)

D7 — when does the gustify script become a suite skill?

Now — spike

The center generator is a gustify-local script. Fastest path, throwaway-priced, lessons gathered on real data.

Gastify window (n=2)

The decision point. Two real projects now exist — what's common becomes the skill, what differs becomes manifest fields. Priced as a rewrite, not a port.

Wave-2 onward

One suite skill (per-project manifest), every project gets a center by running it.

The trade-off both directions, from your own history: promote too early (n=1) = abstracting from a single example — the house rule "defer schema-hood until n≥2" exists because early abstractions lock in wrong guesses. Never promote = gustify's four divergent doc generators, the cautionary tale already in this repo. Impact: purely scheduling — no functionality is lost either way; only rework cost moves. The only decision now: agree the call happens at the gastify window. (Recommended: yes.)
