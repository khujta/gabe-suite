01The investigation in one paragraph

We audited the house the way a plumber audits a leak: not by asking where it feels wet, but by metering every drip. Every usable session transcript from both apps — 36 sessions, 2.5 GB — was digested and read end to end by a 56-agent workflow, which extracted 248 re-instruction incidents: each one a moment where the human had to correct, re-explain, re-route, re-paste, or demand proof. Fable clustered them into 16 patterns, decided the right home for each (skill · command · hook · manifest · deliberately-left-as-judgment), and put the top eight recommendations in front of hostile skeptic agents. Seven survived with modifications. One died. The result is a plan whose first wave is four small steps — and whose first step is protecting work you already paid for.

03The three structural problems

The suite does not have a knowledge problem. The two biggest incident clusters were already written down when they kept happening. What it has is:

1 · Prose doesn't enforce

The E1–E7 execution contract was installed into all 38 command/skill files on 2026-07-02/03 — and the largest cluster (~32 "declared done without looking" incidents) continued after it. In one gastify session the operator had to say "I need you to look for yourself" three separate times. A rule the model must remember is a rule the model will eventually skip; a gate the harness runs is not.

2 · Skills don't fire on ad-hoc asks

The reuse rules live inside gabe-mockup — which loaded 7 times in 5 months. Casual asks ("make a spike for X", "build the reports screen") never route through the skill that contains the rules, so the rules might as well not exist for that work. The operator caught this live: "Use the Gabe mockup skill. We ready-tuned that skill a lot." Dispatch is a first-class fix, not a nice-to-have.

3 · Three divergent suites run under one name

A self-documented "hardening pass" patched 38 files directly in ~/.claude (ledger: ~/.claude/gabe-hardening/APPLY-STATUS.md). It never reached the repo — whose docs site narrates the change its own source lacks — and never reached ~/.agents, which is frozen at the 2026-07-01 install. This is the "endless patching" feeling, mechanized: the patches were fine; the surface was wrong.

⚠ Time-sensitive

A naive ./install.sh re-run today would overwrite the hardened ~/.claude copies with the un-hardened repo versions — destroying paid-for work. Reconciling this is step 0.1 of the plan, before anything else.

06Thread 4 — the lift operation (spike → production, never from scratch)

The signature failure, in the operator's own words: "Why are we trying to recreate everything in the storybook? We already have the layout in the storybook somewhere." Across three arcs (gustify June 6–7, gustify June 27–30, gastify July 1–8) the same loop repeated: build a stand-in instead of the real component → human catches it → the lesson gets encoded → the twin pays for it again, because the lesson landed in project-local files only one twin has.

The fix is one operation with a mechanical pre-flight — shape suite-global, every concrete value from the project's manifest:

Why content alone won't fix it (the skeptic's point)

The hardened skill already contains a reuse ledger and inventory step — and gastify's July incidents happened anyway, because the skill wasn't loaded. So the SOP ships with its dispatch layer: a widened trigger description, a one-line hard rule in each project's CLAUDE.md (the one layer with a measured effect — gustify's P4 incidents stop after that rule landed on 2026-06-30), and optionally a manifest-scoped warn-hook. deliverable 3 §5

07Thread 3 — the Evidence Doctrine, made real

Your locked doctrine — proof scoped by importance, proof form per change type, one living set per feature, evidence doubling as docs — is already 60% built: gabe-execute classifies runtime-journey evidence, gustify has the recorded 3-viewport layout harness, gastify already commits feature-named proof (248 screenshots). What's missing, per the skeptic-corrected design:

PieceDesign (skeptic-corrected)
Importance filterdeclared once, at plan time, by the human — a proof: none|test|visual|journey field per phase. Never judged at gate time by the executing AI (it would classify down)
Proof formsbackend → failing-then-passing test · frontend → side-by-side reference-vs-live · e2e feature → recorded flow (screenshots/GIF; capture tool is a manifest field — pagecast candidate, Playwright+ffmpeg baseline)
Living setONE feature-named folder: proof/<feature>/manifest.json + NN-step.png. Manifests always committed; curated screenshot subset committed via a .gitignore carve-out; dated run-folders terminated as a convention
Enforcementstaged: a deterministic freshness check at the gabe-commit chokepoint (artifact mtime > newest staged change) — WARN-and-LOG for 2–4 weeks, promoted to blocking only where the log shows ignored warnings
Second opiniona fresh-context evaluator (read-only agent) grades the evidence for proof: visual|journey phases — "plausibility is not correctness"
Evidence → docsdoc pages and demos embed proof/<feature>/ artifacts directly — the living-docs principle. This page practices it (see the provenance box up top)

Are we orchestrating too much?

Measured honestly: both. The transcripts hold 155 multi-agent workflow runs across the twins, and the pattern splits clean: orchestration earns its cost for verification — adversarial review passes caught real defects the builder missed — and loses it for taste generation. The receipts: five spike rounds "via delegated agents/workflows" that kept hand-building stand-ins (#138); a design panel + adversarial review that produced an over-engineered multi-state banner when the operator wanted the minimal version (#151); a workflow that shipped 30–55 KB of its own internal critique logs as deliverables (#156); an abstract 4-option mock board no human could judge (#43). Agents multiply a premise; they don't challenge it. The convention answer (Anthropic's own guidance: the simplest thing that works, add agents only where they demonstrably win) becomes plan step 0.5c: before any multi-agent design fan-out, one cheap single-agent spike past the human first — orchestrate to verify, not to generate taste.

Where the suite is AHEAD of the convention

Fairness cut: cwc's durable state is one PROGRESS.md; the suite's .kdbp/ layer (PLAN/LEDGER/DECISIONS/PENDING/RULES/DEPLOYMENTS…) is a far richer implementation of the same principle, and gabe-handoff already is the "agent-maintained handoff" the cwc repo demonstrates. Risk-priced review and the tier system go beyond anything in the reference material. The imbalance is directional, not absolute: over-invested in instruction surface, under-invested in enforcement surface — which is exactly what Waves 0–1 rebalance.

11Operator review round — the docs diet & the code question

Two concerns raised on review (2026-07-08), both now measured and folded into the plan as steps 0.5a and 0.5b.

The docs are out of hand — measured

The twins hold 247 markdown docs under docs/ (gustify 132, gastify 115) plus 144 .kdbp md files — and 80 new md files were created in the last 30 days alone, against a human reading budget of "a couple of documents." The instinct behind this very site is the fix, stated as a convention (plan 0.5b, the documentation diet):

RuleMeaning
Markdown = the agent-facing recordfew, living files, augmented in place. A new md file only for a genuinely NEW topic — never for a new change, iteration, or date (the no-dated-throwaways rule, extended to docs)
HTML = the human-facing layerdiagrams, memes, visual structure per the docsite convention — human reading context is the scarce resource, so the human never has to read the raw record
One hub per project tracks the inventoryevery human-facing doc registered with its source md (the C9 completeness ledger at project level); orphans flagged; the hub is the only required entry point
Enforced, not rememberedgabe-commit docs-audit gains a deterministic warn: new md file outside the allowed homes → "augment the existing topic doc instead." WARN-first, same staging as the evidence hooks
✔ Plan step 0.5b — EXECUTED 2026-07-09 (suite-side)

The docs-audit warn shipped: gabe-commit/scripts/docs-budget.sh WARNs on staged new markdown outside the allowed homes (.kdbp/**, files registered in DOCS.md) and on dated-throwaway filenames — WARN-and-LOG, never blocking. Per-project doc-hub adoption rides each project's freeze window. W1-remainder-log.md

The code question — never audited, now scheduled (and it decides /gabe-simplify)

This investigation mined conversations, not code. Seed measurements taken now say the supply-side question is live: gustify's own CLAUDE.md declares a 500-line-soft / 800-line-hard file discipline, yet ProfileScreens.tsx is 3,805 lines, RecipeFilterPanel.tsx 2,881, RecipeDetailScreen.tsx 2,534, CookingScreenModel.ts 2,460; gastify's prompt_lab/statement/report.py is 3,214 (generated api-types files excluded). Change-locality — Anthropic's "all the modifiable code for a feature in one place" — measures median 7 files per feature commit (of which 2–4 are .kdbp bookkeeping, a suite overhead worth pricing), with a wide tail (p90 16–18), identical across the twins.

✔ Plan step 0.5a — EXECUTED 2026-07-09

The audit ran (deterministic census + one read-only auditor per twin; deliverable 10). Verdicts: gustify material (4 true-monoliths — ProfileScreens holds four unrelated screens; the filter panel took 72 commits in 90 days at 2,881 lines — plus 3 byte-verified duplicated helpers and ~9.9K dead archived lines), gastify moderate-localized (2 true-monoliths; the rest generated, dead-tree, or honestly cohesive). Verdict: the simplify capability goes — as a tiered gate inside gabe-commit (size-budget WARN + triggered quality-only pass, zero new always-loaded surface); the standalone wrapper stays dead. Cleanup backlog → PENDING.md at implementation time.
