01The honest headline

Your system and Anthropic's guidance are not at war — they agree on every principle this investigation cares about: durable state across sessions, verification loops, self-improving artifacts, evidence before "done." Where they part is weight. Their reference implementations carry a fraction of your surface: one progress file where you keep twenty-four, a JSON feature list where you keep narrative prose, git as the ledger where you append a 10,000-line markdown file. The talks sharpen it: Ara's workshop says markdown past ~200 lines is ergonomically bad for agents too — by that line, the suite's biggest state files aren't just unreadable for you; they're expensive for the model that maintains them.

02Fork A — Is KDBP too heavy?

The official long-running pattern is three artifacts: a feature list in JSON (chosen over markdown because it "resists agent modification"), one progress file, and git history as the memory — descriptive commits are the ledger. CLAUDE.md guidance: target under 200 lines. That's the entire durable surface Anthropic ships for the exact problem KDBP solves with ~24 files per project.

The full-minimal option (A3: PROGRESS.md + git only) is named in the record but would destroy the state-table routing and risk-priced PENDING that this investigation measured as working. evidence: A2-migration-log.md

03Fork B — Commands vs. skills: the duality can simply end

The platform moved toward you. Current docs: commands and skills are the same mechanism — skills/deploy/SKILL.md creates /deploy exactly like commands/deploy.md; skills are the superset. And the frontmatter now does natively what the suite hand-rolls: when_to_use (≤1,536 chars) drives automatic invocation — the dispatch fix, native; paths: globs auto-load a skill when matching files are touched — gabe-mockup on **/*.stories.tsx is the Thread-4 trigger with zero prose; hooks: scoped to the skill — the evidence gate ships inside gabe-commit instead of global settings; disable-model-invocation for human-only gates like push; context: fork for isolated satellites; references/ for progressive disclosure.

✔ DECIDED — B2, skills-only (2026-07-09) · EXECUTED 2026-07-09One skill per capability (lean SKILL.md + references/), trigger-sentence when_to_use, paths globs where file context should fire the skill, human-only flags on the shipping gates, skill-scoped hooks for enforcement. Slash invocation survives automatically; the 6 wrapper-shims and 22 command files retire. This absorbs plan step 1.2 (dispatch) and repackages 1.3c (hooks). One caveat carried honestly at decision time: keep the skill bodies host-portable — Codex reads ~/.agents skills but may ignore Claude-specific frontmatter. Resolved by execution: the operator dropped Codex entirely (2026-07-09) — Claude Code is now the only harness, so the caveat no longer applies. evidence: B2-migration-log.md

04Fork C — Orchestration & in-process triggering ✔ decided: recommendation taken · executed suite-side 2026-07-09

Current guidance confirms the plan rather than changing it: default is a single agent + skills; subagents earn their cost for verbose, self-contained, or tool-restricted work; the docs say isolation "may not be worth it for quick tasks." That is plan step 0.5c (orchestrate to verify, not to generate taste) stated officially. Two additions worth taking: the suite's analysis satellites (health, debt, roast, myopic) are natural context: fork skills — isolation without Task-tool ceremony; and the CI-babysitting you still do by hand after push is the canonical /loop-or-routine case from the talks (Sid, Daisy) — consistent with your locked decision #1: mechanical loops go autonomous, design stays visible. Boris's caveat rides along: auto-mode only where test-based verification is already in place — which is exactly what Wave 1.3 builds.

✔ Executed suite-side (2026-07-09)Both additions landed: context: fork frontmatter shipped on gabe-debt/health/roast/myopic (B2), and the /loop CI-babysitting runbook line was added to the PUSH.md template + gabe-push spec (A2). evidence: B2-migration-log.md · A2-migration-log.md
